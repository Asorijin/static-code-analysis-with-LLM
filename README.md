CCI部分原本是分析代码更改意图，需要更改为分析代码执行意图，即这段代码是做什么的。
HV部分是将历史已查明的漏洞嵌入向量数据库，作为静态分析的查询库，不用改动，但是需要找对应语言的已知漏洞，可以去nvd找。需要包括CVE ID，CVE描述等

整体执行流程是：

- HV前置工作：找已知漏洞，输入至CCI组件，让LLM分析上述三个角度，输出为文本，处理为向量数据库(my_build)

- 执行CCI：分析输入的代码执行意图，传输至LLM分析上述三个角度，输出为文本，嵌入为向量
- 执行HV：将上一步嵌入得到的向量在向量数据库中查找（目的是通过代码相似度检出可能的类似漏洞）
将原代码，原代码的CCI分析，HV检索出的相似漏洞，HV检索漏洞的CCI分析整合进prompt，输入至LLM，判定是否为漏洞及判断原因(my_main)

- 执行质量检查，输出指标(check_point)

项目结构：
- my_main:主程序，漏洞检测流程
- my_build_rag:构建向量数据库
- prompt:LLM提示词（需要大改）
- utils:文本分割工具（原本用于去除GitHub提交信息，不适用于本项目） 已移除
- checkout_point:工具指标检测

目前语言默认java，可替换，出于效果考虑，换识别语言建议同步更换数据集语言
